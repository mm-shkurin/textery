# Описание проекта

## Название проекта

Платформа генерации учебных текстов с подписочной моделью.

---

## Описание системы

Необходимо разработать веб-платформу, позволяющую пользователям создавать учебные и деловые тексты на основе пользовательского запроса.

Пользователь должен иметь возможность:
- зарегистрироваться;
- оформить подписку;
- создать реферат, сочинение, доклад, эссе или другой текстовый материал;
- отредактировать полученный текст во встроенном текстовом редакторе;
- просматривать историю своих генераций.

Основной акцент проекта делается на разработке современного цифрового продукта:
- удобного и визуально привлекательного интерфейса;
- продающего пользовательского сценария;
- полноценного текстового редактора для работы с результатом генерации;
- системы подписок;
- продуктовой аналитики;
- сбора маркетинговых данных;
- построения пользовательских воронок.

Проект должен демонстрировать архитектуру современного веб-приложения и подходы к разработке цифровых сервисов с подписочной моделью.

---

# Техническое задание

# 1. Общие требования

Система должна представлять собой веб-приложение с клиентской и серверной частью.

Клиентская часть должна быть реализована на React.

Серверная часть может быть реализована на усмотрение команды с использованием одного из следующих стеков:
- Python;
- Java;
- JavaScript / Node.js.

Приложение должно обеспечивать:
- регистрацию и авторизацию пользователей;
- оформление и управление подпиской;
- создание текстовых генераций;
- редактирование сгенерированных текстов;
- хранение пользовательской истории;
- сбор продуктовой аналитики;
- экспорт аналитических данных.

# 2. Требования к коду

Код должен быть поддерживаемым и расширяемым.

Прикладной уровень как серверной, так и клиентской части должен быть описан в виде набора интерфейсов, где каждый интерфейс - это абстракция над feature (story, пользовательский сценарий).

Каждый пользовательский сценарий должен быть покрыт модульными автотестами.

---

# 3. Требования к регистрации и авторизации

Пользователь должен иметь возможность:
- зарегистрироваться;
- войти в систему;
- выйти из системы;
- просматривать данные своего профиля.

При регистрации пользователя система должна сохранять:
- IP-адрес;
- страну пользователя;
- тип устройства;
- операционную систему;
- язык устройства;
- дату и время регистрации.

Также необходимо сохранять маркетинговые параметры пользователя:
- utm_source;
- utm_campaign;
- utm_content;
- utm_medium;
- utm_term.

Данные должны храниться в базе данных и использоваться для продуктовой аналитики.

Способы входа:
- Через яндекс
- Через гугл
- Код на почту, но код замокать, чтобы не писать интеграцию с почтой

---

# 4. Требования к пользовательскому интерфейсу

Интерфейс приложения должен быть ориентирован на продажу продукта и удобство пользователя.

Необходимо реализовать:
- современный главный экран;
- блоки с описанием возможностей сервиса;
- страницу тарифов;
- экран оформления подписки;
- страницу генерации текста;
- встроенный текстовый редактор для редактирования результата;
- личный кабинет пользователя;
- историю генераций.

Интерфейс должен:
- корректно отображаться на мобильных устройствах;
- иметь современное визуальное оформление;
- обеспечивать простой и понятный пользовательский сценарий;
- минимизировать количество действий до первой генерации.
- акцентировать ценность продукта через продающие блоки, понятные CTA и демонстрацию примеров готовых работ.

---

# 5. Требования к генерации текстов

Пользователь должен иметь возможность:
- выбрать тип работы: реферат, сочинение, доклад, эссе или другой формат;
- ввести тему, требования, объем и дополнительные пожелания;
- запустить генерацию текста;
- получить результат генерации в виде редактируемого документа;
- внести правки во встроенном текстовом редакторе;
- сохранить отредактированную версию;
- просматривать историю генераций.

Генерация текста может быть реализована одним из следующих способов:
- через заранее подготовленные шаблоны и примеры текстов;
- через упрощенную локальную модель или алгоритм генерации;
- через интеграцию с внешним API генерации текста.

Качество генерации текста не является единственной целью проекта.

Основной целью подсистемы генерации является:
- демонстрация пользовательского сценария;
- обработка запросов пользователя;
- хранение результатов генерации;
- возможность дальнейшего редактирования результата;
- интеграция генерации в общий продуктовый сценарий приложения.

---

# 6. Требования к текстовому редактору

Текстовый редактор является одной из ключевых частей продукта.

Редактор должен позволять пользователю:
- просматривать сгенерированный текст в удобном формате;
- редактировать текст после генерации;
- форматировать заголовки, абзацы и списки;
- сохранять изменения;
- возвращаться к ранее созданным и отредактированным материалам.

Редактор должен быть интегрирован в основной пользовательский сценарий: пользователь получает сгенерированный текст и сразу может доработать его без перехода во внешние инструменты.

---

# 7. Требования к подписочной системе

Система должна содержать подписочную модель с возможностью:
- оформления подписки;
- просмотра текущего тарифа;
- включения автопродления;
- отключения автопродления;
- отображения статуса подписки.

---

# 8. Требования к биллингу

Интеграция с реальными платежными системами не требуется.

Необходимо реализовать упрощенный интерфейс биллинга с тестовой реализацией.

Подсистема должна:
- имитировать успешную оплату подписки;
- сохранять информацию об активной подписке;
- сохранять состояние автопродления;
- поддерживать включение и отключение автосписаний;
- хранить историю изменений статуса подписки.

Все данные должны сохраняться в базе данных приложения.

---

# 9. Требования к продуктовой аналитике

Система должна собирать продуктовые события и сохранять их в базе данных.

Необходимо фиксировать:
- посещение сайта;
- регистрацию пользователя;
- авторизацию;
- посещение страницы тарифов;
- начало оформления подписки;
- успешное оформление подписки;
- запуск генерации;
- успешную генерацию;
- открытие текстового редактора;
- сохранение отредактированного текста;
- отключение автопродления.

---

# 10. Аналитические представления и отчеты

В базе данных необходимо реализовать аналитические представления (view) для построения продуктовых отчетов.

Необходимо реализовать:
- базовые продуктовые воронки;
- агрегированные аналитические данные;
- выгрузку аналитических данных в CSV.

---

# 11. Примеры продуктовых воронок

## Воронка регистрации

1. Посещение сайта
2. Переход к регистрации
3. Успешная регистрация
4. Первая генерация текста
5. Первое сохранение текста в редакторе

---

## Воронка подписки

1. Посещение страницы тарифов
2. Начало оформления подписки
3. Успешное оформление подписки
4. Повторное использование сервиса

---

# 12. Требования к хранению данных

В базе данных необходимо хранить:
- пользователей;
- подписки;
- события аналитики;
- данные текстовых генераций;
- отредактированные версии текстов;
- маркетинговые параметры;
- технические параметры устройств пользователей.

---

# 13. Ожидаемый результат

По итогам разработки должна быть создана демонстрационная веб-платформа с:
- современным продающим пользовательским интерфейсом;
- подписочной моделью;
- системой продуктовой аналитики;
- пользовательскими воронками;
- системой генерации рефератов, сочинений, докладов и других текстов;
- встроенным текстовым редактором;
- хранением пользовательских данных и аналитики.
