## Требующие проверки перед действием (нужен ручной прогон/подтверждение)

8. **Дублирующиеся CI-workflow** (`.github/workflows/backend-ci.yml` vs `backend/.github/workflows/ci.yml`; аналогично для frontend). Они реально разошлись по подходу (service-контейнеры vs apt-get install postgres/redis, разные пароли БД). Перед объединением нужно выяснить, какой вариант реально проходит на используемом CI-раннере (в git-истории видно, что серия фиксов 10.07 была вызвана именно проблемами раннера — возможно, apt-get-подход появился как обходной путь для конкретного раннера, у которого нет доступа к Docker socket/systemd). Решение — не чисто техническое, а зависящее от инфраструктуры CI-раннера, которую этот аудит не проверял вживую.
9. **Redis-сервис в `infra/docker-compose.yml` и в CI**, который не используется кодом. Прежде чем убирать — подтвердить, что в ближайших планах действительно нет перехода на `arq` (если план есть — оставить как задел; если нет — убрать как мёртвый провижининг).
10. **`backend/adapters/db` тесты** не были прогнаны в рамках аудита (нужен реальный Postgres на localhost:5432). Перед тем как полагаться на "готово", стоит прогнать их против чистой БД (например через `infra/docker-compose.yml`, если это не создаст конфликт с другими сессиями на хосте).
11. **Acceptance-тесты (backend+frontend)** также не прогонялись — нужен полный стек (БД, backend-процесс, собранный frontend, ChromeDriver). Перед тем как объявлять Story 1 "рабочим вертикальным срезом", стоит один раз прогнать полный `acceptance/` набор целиком.
12. **Финальный re-audit `backend-audit-remediation.md`** после Task 1+2 — документ сам признаёт "Not yet re-audited after remediation". Стоит либо провести финальный проход по той же методологии и записать результат, либо явно закрыть документ с пометкой, что переаудита целиком не будет.

## Архитектурно значимые (требуют решения пользователя, не просто правки текста)

14. **`infrastructure/` (пустая) vs `infra/` (реальная)** — это самое значимое расхождение всего аудита. Весь корневой `.claude/rules/infrastructure.md`, `.claude/guidelines/infrastructure-detail.md` и скиллы (`/run-backend`, `/stop-backend`, `/test-acceptance`, `/demo`, `agent-logging.md`) ссылаются на несуществующий каталог `infrastructure/`. Решение пользователя — один из двух путей:
    - **(а)** Создать реальный `infrastructure/` и перенести туда содержимое `infra/` (включая `infra/.claude/`, docker-compose, Dockerfile'ы, `.env`) — тогда корневой фреймворк заработает по инструкции буквально.
    - **(б)** Переписать все ссылки в правилах/guidelines/skills корневого `.claude/` с `infrastructure/` на `infra/` — тогда не нужно ничего физически переносить.
    - В любом случае — это не мелкая правка, а решение, затрагивающее сразу несколько always-on файлов фреймворка, поэтому не должно делаться попутно с другой работой.
15. **`infra/.claude/` — отдельный мини-фреймворк ("Mini Dev-Loop Harness")**, частично адаптированный под Textery (реально используется `python-fastapi.md`/`react-ts.md`/architecture.md), но содержащий существенный балласт неиспользуемых generic-агентов (android, ios, flutter, terraform-yandex, mobile, devops — 6+ файлов, не относящихся к текущему стеку проекта). Решение пользователя: оставить как есть (задел на будущее — вдруг появится мобильное приложение или Terraform), или вычистить неиспользуемые агенты, чтобы не запутывать будущих читателей.
16. **`infra/Makefile`** содержит только Terraform-цели, хотя `infra/architecture.md` прямо говорит "No cloud/Terraform involvement yet (dormant)". Нужно решение: либо это осознанный задел, либо шаблонный мусор, который стоит убрать до момента, когда Terraform реально понадобится.
17. **`/cleanup-chrome` skill vs `.claude/rules/infrastructure.md`** — прямое текстовое противоречие (см. `04_DOCUMENTATION_DRIFT.md` пункт 19). Требует решения: либо скилл получает более безопасный механизм остановки (например убивать только процессы, порождённые текущей сессией/тестовым прогоном, не по имени exe целиком), либо правило `infrastructure.md` получает явное исключение для этого конкретного случая.
18. **`.memory-bank/` (корневой) и `infra/.memory-bank/`** — две отдельные, ныне замороженные системы записей, дублирующие часть функции `ProductSpecification/`. Решение пользователя: официально задокументировать их как "исторический входной материал, не поддерживается активно" (кроме `known-debt.md`, на который продолжают ссылаться активные документы), либо периодически синхронизировать, либо архивировать целиком.

## Потенциально опасные (только после явного подтверждения пользователя, не трогать по умолчанию)

19. **`qodana.yaml` в корне** — untracked черновик с линтером `jetbrains/qodana-jvm-community` (для JVM, не для Python/TypeScript этого проекта). Если файл коммитить как есть и подключать к CI — статический анализ не будет работать корректно ни для одного из двух реальных языков стека. Не удалять автоматически (возможно, пользователь сам сейчас экспериментирует с этим файлом) — но перед коммитом стоит заменить линтер на подходящий (`qodana-python`/`qodana-js` или community-варианты под оба языка) и решить, действительно ли Qodana нужен проекту (в документации решение о её использовании нигде не зафиксировано).
20. **`.idea/` в корне** — стандартные файлы IntelliJ/PyCharm. Проверено: уже корректно исключён в `.gitignore` (`.idea/` — первая строка) и не отслеживается git (`git ls-files .idea` — пусто). Действий не требуется, упомянут только для полноты карты проекта.
21. **`.agents/skills/caveman*`/`cavecrew`** — сторонний плагин пользователя, установленный из внешнего GitHub-репозитория (`skills-lock.json`). Он не связан с продуктом Textery и не мешает работе фреймворка `.claude/`, поэтому это не мусор, требующий удаления, а личный инструмент. Упоминаю здесь только для полноты картины — решение оставлять/убирать целиком на усмотрение пользователя, аудит не находит в этом проблемы.
