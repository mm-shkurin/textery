# Справочник по агентному фреймворку Claude в проекте Textery

> Read-only аудит. Ничего в `.claude/`, `.agents/`, `infra/.claude/` не менялось.

## 0. Что это вообще такое

Это не просто "Claude помогает писать код" — это **шаблон-конвейер для разработки методом TDD** (test-driven development, "сначала тест, потом код"), где почти каждый шаг выполняет не пользователь руками, а специализированный "саб-агент" — отдельный вызов Claude с узкой ролью (написать один падающий тест / написать минимальный код / проверить строгость проверок / почистить код и т.д.).

Смысл в трёх словах: **правило → делегирование → память в файлах**. Правила лежат в `.claude/rules` и `.claude/guidelines`, конкретные роли — в `.claude/agents`, кнопки для человека — в `.claude/skills`, а поскольку у Claude нет памяти между разговорами, всё важное состояние (что сделано, что дальше) хранится не "в голове", а в файлах `progress.md`.

Важно: этот фреймворк **написан как переиспользуемый шаблон**, не специфичный для Textery — правило об этом прямо зафиксировано в `.claude/guidelines/prompt-rules.md`. Конкретика про Textery (какая база данных, какой AI-провайдер, как называется история) живёт отдельно, в `ProductSpecification/technology.md` и `ProductSpecification/stories/`.

## 1. Слои документации — от общего к частному

| Слой | Путь | Когда загружается | Роль (аналогия) |
|---|---|---|---|
| Точка входа | `CLAUDE.md` (корень) | Всегда | Оглавление |
| Always-on правила | `.claude/rules/{coding-rules,infrastructure,workflow}.md` | Всегда, в каждой сессии | Конституция — короткая, обязательная |
| Guidelines (детали) | `.claude/guidelines/*.md` | По требованию — только "перед тем как делать X" | Отраслевые своды правил |
| Агенты | `.claude/agents/*.md` | Вызываются инструментом `Agent` | Должностная инструкция одной роли |
| Скиллы | `.claude/skills/*/SKILL.md` | Вызываются командой `/имя` | Кнопка на пульте — тонкий диспетчер |
| Шаблоны | `.claude/templates/*` | Подгружаются агентами по пути | Готовые бланки/формы |
| Тех-профили | `.claude/tech/{profile}/*` | Резолвятся через `technology.md` | Отраслевой стандарт под конкретный стек |

Идея разделения на "always-on" и "по требованию": не держать в контексте Claude сразу все детали (это дорого по токенам), а подгружать только то, что нужно для текущего шага.

## 2. Always-on правила (`.claude/rules/`)

- **`coding-rules.md`** — Clean Architecture (`domain → usecase → adapters → application`, зависимости только внутрь), запрет вызывать один usecase из другого, лимит 200 строк на любой файл (включая тесты и конфиги).
- **`infrastructure.md`** — вся инфраструктура должна управляться только через файлы в каталоге `infrastructure/` (IaC — "инфраструктура как код"), никаких ручных правок на серверах. Плюс защитные правила против того, чтобы одна параллельная Claude-сессия сломала другую на общем компьютере (не убивать процессы по имени, не трогать чужие Docker-контейнеры).

  **⚠️ Важно: каталог `infrastructure/`, который описывает это правило, физически пуст в проекте.** Реальная инфраструктура лежит в `infra/` (без "structure"). Подробности — в разделе 8 и в `04_DOCUMENTATION_DRIFT.md`.

- **`workflow.md`** — жизненный цикл истории (интервью → спецификация → backend → integration → frontend → security → load → infra сценарии), формат `progress.md`, правило "атомарной рабочей единицы" (нельзя останавливаться посреди цепочки саб-скиллов).

## 3. Guidelines — детали по темам (`.claude/guidelines/`)

| Файл | О чём |
|---|---|

| `frontend-rules.md` | Паттерн "тонкий компонент + вся логика отдельно", запрет хардкодить плейсхолдеры из мокапов, Selenium-конвенции (см. ниже), правила для CSS. |

## 4. Скиллы — 33 команды, сгруппированы по частоте использования

**Ежедневные, ядро TDD-цикла:**
- **`/continue`** — главный оркестратор. Читает `progress.md` истории/задачи, находит следующий незавершённый шаг, вызывает нужных саб-агентов по очереди, коммитит результат. Это самая часто вызываемая команда в системе — именно её обычно печатает разработчик, чтобы "продолжить, откуда остановились".
- **`/test-review`** — после написания теста ищет "слабые" проверки (например `contains`/`isNotNull` вместо точного сравнения) и чинит их.
- **`/refactor`** — чистит код после green-фазы (структура, без изменения поведения). Всегда отдельным коммитом от основной работы.
- **`/test-coverage`** — считает покрытие после green-фазы, если находит недостигнутую, но достижимую ветку логики — сама добавляет в `progress.md` новые red/green шаги.
- **`agent-review`/`premortem`** — не ручные команды, а автоматические "два свежих взгляда" перед финальным коммитом рабочей единицы (см. раздел 6).

**Редкие, по одному разу на историю (в начале):**
`/interview` (структурированное интервью с пользователем → `interview.md`) → `/story` (спецификация) → `/mockups` (HTML-мокапы desktop+mobile) → `/api-spec` (OpenAPI) → `/test-spec` (BDD-тесты в 6 категориях).

**Задачи и восстановление контекста:**
- **`/task`** — создать задачу типа bug/refactoring/qa.
- **`/rca`** — доказательный поиск причины бага (никогда не принимает "правдоподобную версию" без реальных логов/тестов).
- **`/handoff`** — **единственный** писатель "дневника" (journey summaries) и `carryover.md` — пишется только когда случилось что-то нетривиальное, не рутинно.
- **`/qa-run`** — ручной чек-лист против prod-copy окружения, только через клики в UI.

**Frontend-специфичные:** `/align-design`, `/design-review`, `/screenshot`, `/demo`.

**Инфраструктурные утилиты:** `/run-backend`, `/run-frontend`, `/stop-backend`, `/test-acceptance`, `/test-adapter`, `/test-usecase`, `/test-frontend`, `/test-all`, `/cleanup-chrome`.

**Мета-документация:** `/prompt-update`, `/prompt-refactor`, `/doc`, `/architecture`, `/design-preview`.

**Разное:** `/plain` — кнопка "объясни просто", которую сам агент никогда не нажимает по своей инициативе — только по явной просьбе пользователя.

## 5. `progress.md`, `carryover.md`, summaries, ADR — простыми словами

У Claude нет памяти между разговорами (сессия обнуляется командой `/clear`). Вся "память" проекта — в закоммиченных файлах:

- **`progress.md`** — чек-лист с галочками `[x]`(готово) / `[~]`(в процессе) / `[ ]`(не начато) / `[S]`(пропущено). Как список дел на холодильнике: новая сессия открывает файл и сразу видит, что делать дальше. Это именно то, что делает работу возобновляемой без пересказа всей истории заново.
- **Summaries** (`stories/…/summaries/{сценарий}.md`) — "дневник неожиданностей": пишется только когда предсказание не сбылось, было принято архитектурное решение, или код повёл себя не так, как ждали. Пишет их **только** `/handoff`. Если ничего необычного не было — файл не создаётся вовсе, и это нормальный исход.
- **`carryover.md`** — выжимка самых долгоживущих "хитростей" из summaries, когда история/задача уже полностью закрыта — чтобы следующая работа не читала все старые дневники по отдельности.
- **ADR** (`decisions/*-decision.md`) — короткий документ "почему выбрали именно такое архитектурное решение и что отвергли". Пишется только для неочевидных решений. Для Story 1 их два: `persist-and-enqueue-architecture-decision.md`, `request-validation-architecture-decision.md`.

## 6. RED → GREEN → review passes: как всё связано внутри одной "рабочей единицы"

1. **RED-агент** пишет один тест, который должен упасть — сначала письменно предсказывает, как именно (тип ошибки, сообщение), запускает, сверяет предсказание построчно с реальностью. Несовпадение = предсказание неверное, дальше разбираться, а не "делать вид, что всё в порядке".
2. **`/test-review`** — чинит слабые проверки в только что написанном тесте.
3. **`/refactor`** — чистит тестовый код (отдельный коммит).
4. **GREEN-агент** — снимает пометку "тест отключён" и пишет минимальный код, чтобы тест прошёл. Ему запрещено менять сам тест (кроме снятия пометки).
5. **`/test-coverage`** — проверяет покрытие после green, может добавить новые red/green шаги в `progress.md`.
6. **`/refactor`** снова — чистит production-код.
7. Непосредственно перед финальным коммитом рабочей единицы, в одном "заходе" с рефактором, запускаются **`agent-review`** ("что здесь не так — баги, нарушенные инварианты") и **`premortem`** ("представь, что это уже сломалось в проде — чего не хватает"). Оба **не блокируют** коммит — это подсветка проблем в отчёте, а не обязательный гейт (в отличие от `test-review`/`refactor`, которые обязательны).

## 7. Технологический профиль (двухслойная загрузка)

`ProductSpecification/technology.md` объявляет 4 ключа: `backend`, `frontend`, `css`, `browser-testing`. Для Textery: `python-fastapi-hex`, `react-ts`, `plain-css`, `selenium`. Каждый резолвится в свою папку `.claude/tech/{значение}/`. Смысл: универсальные правила не привязаны к языку — один и тот же набор агентов работает для Java, Python, Node и т.д., просто подставляя нужный тех-профиль. В `.claude/tech/` заготовлено around 12 backend-профилей — подтверждение того, что это переиспользуемый шаблон, из которого Textery использует лишь малую часть.

## 8. Главная системная аномалия: `infrastructure/` vs `infra/`

Это самая важная находка по фреймворку.

- **`infrastructure/`** (без "i" перед "structure" — то есть полное слово) физически пуст (0 файлов, 0 коммитов в git-истории).
- Весь корневой `.claude/rules/infrastructure.md` + связанные guidelines + скиллы (`/run-backend`, `/stop-backend`, `/test-acceptance`, `/demo`, `agent-logging.md`) ссылаются именно на `infrastructure/.env`, `infrastructure/scripts/*.sh`, `infrastructure/agent-progress.log`, `infrastructure/notes/*.md` — ни один из этих путей не существует.
- Реальная инфраструктура (Docker Compose, Dockerfile'ы, nginx) лежит в **`infra/`** — это подтверждено явно в `ProductSpecification/technology.md` ("provisioned by the separate `infra/` harness").
- Доказательство реального сбоя: `infra/agent-progress.log` содержит настоящие, свежие логи от корневых TDD-агентов (red-agent, test-review-agent, refactor-agent и т.д.), в точности соответствующие формату из `agent-logging.md` и реальным коммитам git-истории — а `infrastructure/agent-progress.log` пуст.

**Вывод:** `.claude/` — это переиспользуемый шаблон, изначально рассчитанный на каталог `infrastructure/`. Команда Textery вместо этого создала отдельный, самостоятельно эволюционировавший `infra/` (со своим мини-фреймворком, см. ниже) — и никто не обновил ссылки в правилах/скиллах корневого фреймворка. Практическое следствие: skills `/run-backend`, `/stop-backend`, `/test-acceptance`, `/demo` не будут работать по написанной в них инструкции буквально — но на практике, судя по всему, "выживают" за счёт того, что реальные команды (`uvicorn ...`, `npm run dev`, `pytest backend/`) продублированы напрямую в `ProductSpecification/technology.md` в разделе Conventions.

## 9. `infra/.claude/` — отдельный, но НЕ заброшенный мини-фреймворк

Внутри `infra/` есть собственный, отдельный от корневого набор agents/skills/hooks под названием **"Mini Dev-Loop Harness"** (см. `infra/AGENTS.md`), нацеленный именно на deployment-задачи (Docker Compose и т.п.), а не на TDD-цикл продукта:

- Скиллы: `infra:plan` (планирование фичи через planner+skeptic саб-агентов), `infra:review` (ревью против плана), `infra:debug` (поиск причины бага в deployment-слое).
- Агенты (`infra/.claude/agents/*.md`) — это generic-набор под ЛЮБОЙ стек: android, ios, flutter, terraform-yandex, node-ts, devops, mobile, backend, frontend, planner, python-fastapi, react-ts, reviewer, skeptic. Из них Textery реально использует только `python-fastapi.md` и, вероятно, `react-ts.md`/`frontend.md` — остальные 10+ (android/ios/flutter/terraform-yandex/mobile/devops) представляют собой неиспользуемый балласт скопированного целиком шаблона.
- `infra/architecture.md` — не шаблонный, реально написан под Textery (Postgres+Redis+FastAPI+React/nginx, конкретные пути Dockerfile, результаты валидации `docker compose up -d --build`).
- `infra/Makefile` — содержит только Terraform-цели, хотя `infra/architecture.md` прямо говорит "No cloud/Terraform involvement yet (dormant)" — то есть шаблонный остаток, не адаптированный под текущее состояние проекта.
- `infra/.claude/settings.json` подключает Stop-хук `test-gate.sh`, который блокирует завершение сессии, если агент правил код, но ни разу не запустил тесты — механизм защиты, работающий иначе, чем в корневом фреймворке (через hook, а не через промпт-инструкцию).

Конфликтов имён скиллов/агентов с корневым `.claude/` нет — но оба фреймворка независимо претендуют на "инфраструктурную" территорию, и именно эта путаница (`infra` vs `infrastructure`) — вероятная причина сбоя из раздела 8.

## 10. `.agents/skills/` (caveman*, cavecrew) — сторонний плагин, не о Textery

Подтверждено `skills-lock.json`: все 7 скиллов (`caveman`, `cavecrew`, `caveman-commit`, `caveman-compress`, `caveman-help`, `caveman-review`, `caveman-stats`) установлены как маркетплейс-плагин из внешнего GitHub-репозитория `JuliusBrussee/caveman`. Это личный набор инструментов пользователя для сверхсжатого стиля общения с Claude (экономия токенов), генерации терсе commit-сообщений и т.п. — никак не упоминается в `.claude/rules`, `.claude/guidelines` или `ProductSpecification/`, никак не участвует в разработке продукта Textery.

## 11. Признаки параллельной работы нескольких сессий/людей

- Git-история подтверждает минимум двух авторов коммитов (`feodrjob`, `mm shkurin`) и несколько feature-веток (`agent-frontend`, `evening-demo-backend`, `features/story-1-auto-generate-doklad`, `features/story-1-frontend`) — все полностью слиты в `dev` (см. `03_REAL_PROJECT_STATUS.md`, раздел Git).
- Коммит `b5ac291` ("docs: record git-worktree pattern for parallel sessions on one story") прямо документирует практику: backend-сессия работает в основной рабочей копии, frontend-сессия — в отдельном git worktree, обе сливаются в `dev` независимо. Это подтверждено и структурой git-графа (два параллельных потока коммитов, слитых отдельными merge-коммитами в один и тот же период).
- `.claude/guidelines/prompt-rules.md` прямо признаёт этот факт как норму работы над проектом, а не как исключение.
- Найдено внутреннее противоречие: skill `/cleanup-chrome` рекомендует `taskkill /IM chrome.exe /F` (убийство процессов по имени исполняемого файла) — это прямо нарушает always-on правило `.claude/rules/infrastructure.md` ("NEVER kill processes by executable name... kills ALL instances system-wide, including other Claude sessions"). Skill предупреждает пользователя о риске для его собственного браузера, но не о риске для параллельных Claude-сессий на этой же машине.

## Практический вывод

Фреймворк реально используется и работает (это подтверждается десятками TDD-коммитов, реальными `progress.md`/summaries/ADR по Story 1) — но содержит одно системное несоответствие путей (`infrastructure/` vs `infra/`), которое стоит либо исправить в правилах, либо создать `infrastructure/` как ожидается — это решение пользователя, не то, что можно исправить в рамках аудита.
