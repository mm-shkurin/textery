# План: прогнать все фичи через TDD-фреймворк (backfill off-framework работы)

Составлен 2026-07-20. Причина: по пятницам под дедлайн часть фич шла **вне** цикла
red→green→refactor (evening-demo слайс, модалки, manual-mode фронт, часть auth-обвязки).
Цель — довести всё до полного соответствия фреймворку, не переделывая то, что уже
реально покрыто.

## Принцип (что НЕ делаем и что делаем)

**Не** перезапускаем фреймворк на коде, который уже честно покрыт. Большинство `[S]` в
`progress-*.md` — легитимные пропуски цикла («нет порта для инъекции», «поведение уже
покрыто прошлым сценарием», «guard уже существует»). Их трогать не нужно — это фреймворк,
работающий правильно.

Целим ровно в 3 типа дыр, где фреймворк реально пропущен:

1. **Код без red→green родословной** — построен руками, `[x]`/`[S]` описывают поведение,
   которое никто не проверял тестом (known-debt #10).
2. **`[S]` / `@pytest.mark.skip` с ныне-ложной причиной** — самый опасный: причина устарела,
   тест не перезапускается, e2e-доказательства нет, а чекбокс зелёный.
3. **Отсутствующая acceptance-нога (e2e)** — unit/adapter есть, но реального прохода через
   границу (сервер/браузер) нет.

Правило: **зелёный `[x]` на off-framework коде — ложь, пока нет red→green или acceptance.**

## Фазы работ (порядок по риску)

### Фаза A — Reconcile-развёртка (дёшево, без прод-кода)
Для каждой истории: собрать **все** `[S]` и `@pytest.mark.skip`, классифицировать каждый:
- `legit` — делать нечего (нет порта / покрыто прошлым сценарием) → оставить;
- `deferred` — поведение отложено, ещё не построено → в backlog Фазы C;
- `stale` — причина ложна сегодня → в backlog Фазы B.

Выход — чеклист только `deferred`+`stale`. Это реальный объём backfill'а. Зелёным
чекбоксам не доверять до сверки. Команда: `grep -rn "\[S\]\|pytest.mark.skip" <story>`.

### Фаза B — Закрыть stale-skip дыры (e2e-доказательства)
Un-skip + сверка причины + зелёный. Уже известные две:
- `test_verify_valid_code_acceptance.py` — снять skip; если 3.2 rejection ещё нет, тест
  вскроет реальный гэп (это и есть цель), затем закрыть вместе с Фазой C.
- `test_manual_editor_acceptance.py` — снять skip, поправить локатор под contenteditable,
  зелёный.
- landing hero-subheading (known-debt #12) — убрать ассерт или вернуть subheading.

### Фаза C — Достроить недостающее поведение через настоящий red→green
Полный work-unit на сценарий (`red-* → /test-review → commit → /refactor → commit`, затем
`green-* → /test-coverage → commit → /refactor → commit`), по одному сценарию, через
`/continue`:
- **Story 7 rejection-ветки 3.2–3.6** — `matches()` else-ветка, `Optional`-разыменования
  (неизвестный email → чистый 404/400, не 500).
- **Story 5 backend** — сначала забутстрапить `progress-backend.md` из
  `tests/{01_API,05_Security,06_Integration}_Tests.md`, затем TDD по сценариям.
- **Story 1 остаточные `[ ]`**, где код уже что-то делает → характеризационный red
  первым (как Task 3), потом сверить `[x]`.

### Фаза D — Закрытие внешних аудитов
- `backend-audit-remediation.md` (baseline 2.0/3.0) и `frontend-audit-remediation.md`
  (2.5/3.0) — либо финальный проход по методологии, либо явно закрыть с остаточным списком.

## Процессная защита (чтобы пятница не плодила долг заново)

1. **Off-framework под дедлайн — можно, но с уговором:** тем же коммитом завести
   backfill-задачу (образец — Task 3) со списком точных сценариев к перекрытию.
2. **Чекбоксы:** off-framework код метить `[S]` c причиной «off-framework, backfill
   pending» — **никогда `[x]`**. `[x]` только после red→green или acceptance.
3. **Skip-причины датировать** — `@pytest.mark.skip(reason="RED 2026-07-DD: ...")`.
   Фаза A ловит те, чья причина протухла.
4. **Еженедельно** (начало спринта) — прогон Фазы A по всем активным историям + полный
   `acceptance/` набор против живого стека (один раз, как в `CLEANUP_PLAN` фаза 7).

## Рекомендованный первый шаг

Фаза A по story 5 и 7 (самые свежие, больше всего `[S]`), затем Фаза B (2 known
stale-skip) — дёшево, сразу возвращает e2e-доверие. Потом Фаза C story 7 rejection-ветки
(единственная реальная дыра безопасности).
