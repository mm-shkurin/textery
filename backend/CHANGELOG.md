# Changelog

Формат — [Keep a Changelog](https://keepachangelog.com/ru/1.1.0/),
версии — [SemVer](https://semver.org/lang/ru/).

Пока API не объявлен стабильным, минорная версия меняется на каждый закрытый
срез (story), патч — на исправления и внутренние изменения. Версия живёт в
`pyproject.toml` и должна совпадать с верхним заголовком этого файла.

## [Unreleased]

### Added

- Статическая проверка типов: `mypy` над всем production-кодом и тестами
  `domain`/`usecase` (196 файлов), отдельным джобом `types` в CI. Ранее
  аннотации были во всех сигнатурах, но их никто не проверял. 15 statements-
  модулей внесены в закрытый список исключений — новый тестовый модуль
  проверяется с первого коммита.
- Порог покрытия в CI (`--cov-fail-under=90`). До этого `coverage.xml`
  выгружался артефактом и ни на что не влиял.
- Юнит-тесты на `GetDocument`, `ListDocuments`, `GetGeneration`,
  `ListGenerations` — раньше их гоняли только через router-тесты, где сам
  usecase подменён стабом.

### Changed

- `RegistrationResult.verification_code` больше не `Optional`: `RegisterUser`
  такого состояния не порождает, а тип заставлял DTO читать поле у значения,
  которое по типу могло быть `None`.
- Composition root: 13 фабрик с одинаковым блоком «открыть сессию — отдать
  usecase — закрыть в `finally`» свёрнуты на `request_scoped`.

### Fixed

- Очистка тестовой БД (`TRUNCATE`) не включала таблицы `oauth_*`, из-за чего
  Postgres отклонял операцию целиком, строки утекали между тестами и полный
  прогон давал 8 падений и 45 ошибок.
- Набор `adapters/db` без поднятого Postgres зависал вместо понятного `skip`.
- Джоб `lint` в CI был красным: 21 замечание `ruff check` и 19 файлов без
  форматирования.
- `FakeTokenService` не реализовывал `read_access_subject`: метод добавили в
  порт `TokenService` ради Bearer-зависимости, а дубль не обновили — он тихо
  перестал быть реализацией порта.

## [0.1.0]

Первый срез MVP: регистрация, вход, генерация документа через GigaChat и
ручной редактор. Всё под Clean Architecture — `domain → usecase → adapters →
application`, зависимости только внутрь.

### Added

- **Аутентификация по email.** Регистрация с кодом подтверждения, повторная
  отправка кода с кулдауном, подтверждение аккаунта, вход, обновление пары
  токенов. Пароли — bcrypt, токены — JWT HS256 с раздельными типами `access` и
  `refresh`.
- **Защита от перебора.** Счётчик подряд идущих неудачных входов, блокировка
  аккаунта на пороге, сброс счётчика при успешном входе. Инкремент атомарен на
  стороне БД.
- **Вход через Яндекс OAuth.** Legs `start` / `callback` / `exchange`, привязка
  CSRF-state, одноразовый handoff-код вместо токена в редиректе,
  общий на все реплики rate-limit по (leg, источник).
- **Генерация документов.** Постановка задачи, фоновое исполнение с
  экспоненциальным backoff и джиттером, периодический sweep зависших генераций
  после перезапуска процесса.
- **Ручной режим.** Создание документа, идемпотентное по `Idempotency-Key`,
  чтение, сохранение с оптимистичной блокировкой по `version` (single-statement
  compare-and-swap), санитизация HTML через nh3.
- **История с keyset-пагинацией** для генераций и документов — курсор по
  неизменяемой паре `(created_at, id)`.
- **Единый конверт ошибок** `{error_code, message}` для всех ответов, включая
  404 и 500; внутренние детали уходят только в лог.

### Security

- Все данные привязаны к владельцу, предикат `owner_id` — в SQL. Чужой ресурс
  отвечает `404`, а не `403`: `403` подтвердил бы, что он существует.
- `JWT_SECRET` без значения по умолчанию и с проверкой длины ≥32 байт
  (RFC 7518 §3.2) на старте; `DATABASE_URL` — именованная ошибка конфигурации.
- Вход не различает «нет такого аккаунта» и «неверный пароль»; проверка
  `is_verified` идёт после проверки пароля, чтобы ответ не работал оракулом
  существования аккаунта.

### Known limitations

- Refresh-токены stateless, списка отзыва нет: утёкший refresh-токен работает
  до истечения срока.
- Очередь генераций (`NoOpGenerationQueue`) не переживает перезапуск процесса;
  восстановление держится на sweep с интервалом 60 с.
