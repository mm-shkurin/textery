## Fix: dirty flag only set by the DOM input handler, never by programmatic transactions
- [x] root cause analysis — confirmed by probe during Story 5 scenario 7.9's premortem: `setHasUnsavedChanges(true)` has one call site, `editorProps.handleDOMEvents.input` (`ManualEditor.tsx:117-118`); programmatic toolbar transactions emit no DOM `input` event. See spec.md for the probe output and the `hasUnsavedChanges` init-to-`true` trigger detail.

See spec.md "Fix approach" for the three corrections.

- [ ] red-frontend-inflight — the twin five groups converged on: a **toolbar** action during an in-flight save must queue a re-save. Assert a second `saveDocument` fires carrying the marked-up HTML at the server-returned version, and the status never reads `Сохранено` while unsent content exists. `ManualEditor.saveStatus.test.tsx:125` is the right shape but drives `fireEvent.input`, so it stays green through this regression.

  **+ pre-load window guard (premortem BLOCK on `fe8bf29`, folded in here rather than deferred — this step is the one that installs the active-clean write, so the guard belongs with it).** Nothing gates the editor while `getDocument` is in flight: `ManualEditor.tsx:177` renders `<EditorContent>` unconditionally, `useEditor` sets no `editable: false`, initial `content: ''` (`:111`). The visitor sees an empty, fully typeable editor and can't save their way out — `handleSave` early-returns on `!documentId` (`:45`), and `documentId` is set by the same `.then()` that calls `setContent` (`useDocumentInit.ts:32`) and clobbers what they typed. **Today this is bad but honest** — `hasUnsavedChanges` inits `true`, so the status still reads `Черновик` after the clobber. **After this task's green it goes silent**: spec.md's "Fix approach" §3 puts an active `setHasUnsavedChanges(false)` in that same `.then()`, after the clobber — the editor eats the text and confidently reads `Сохранено`. That is the headline bug relocated onto the load path by its own fix, and it violates spec.md's stated "unknown origin → dirty" fail-safe. No test in the suite holds `getDocument` pending and types (`ManualEditor.reopen.test.tsx:31-39` and all six `*.parseHTML` files await the load first); the 8-group scan covered the load *direction* but assumed the load lands before the user exists. **Guard:** render with `existingDocumentId`, `getDocument` returning a manually-held pending promise, `fireEvent.input` typed content into `editor-content-area`, *then* resolve — assert the status is NOT `Сохранено`, and pin whether the typed text survives `setContent`. **Second trigger, same guard:** `useDocumentInit`'s effect deps are `[documentType, existingDocumentId]` (`:54`), so a mid-session prop change re-runs the clobber-plus-active-clean against a document with unsaved edits; drive the same assertion via a prop change.

  Premortem dismissed with reasons (do not re-litigate): the typing→dirty path is already guarded by `saveStatus.test.tsx:38-59` and `:125`, both of which go red if green breaks it; undo-past-the-load is real but pre-existing and outside this fix's causal chain (history behaviour is identical before and after, and `onUpdate` marks it dirty correctly); `horizontalRule`'s ghost-filler can't fire (`isRequired: true` disqualifies it from `defaultType`) and would be fail-safe if it did; Placeholder is decoration-only, no `docChanged`; collapsed-selection `toggleMark` writes only `storedMarks`, no doc change, correctly stays clean.
