## Frontend — что построено (2026-07-15)

Ручной режим создания документа (без LLM) теперь живёт как полноценный флоу на фронте,
параллельно с существующим auto-режимом. Модалка выбора режима больше не показывает
ручной режим как "скоро" — обе карточки кликабельны. Выбор ручного режима создаёт
пустой документ на бэке и сразу открывает редактор — без skeleton, без ожидания:
пустая content-область с плейсхолдером и полный тулбар (заголовки h1/h2, абзац,
список, bold, italic) видны сразу.

Редактор построен на Tiptap. Форматирование (bold и т.д.) применяется через тулбар
или сочетания клавиш, и активная кнопка тулбара отражает форматирование именно в
точке курсора — не глобальное состояние документа. Это работает как для выделения
текста, так и при простом перемещении курсора между отформатированным и обычным
текстом (курсорное отслеживание через isActive + перехват selection-события).

Сохранение — через PUT на бэк с optimistic-concurrency (version). Пока сохранение
идёт, кнопка "Сохранить" блокируется и показывает спиннер. Если пользователь дёргает
сохранение повторно, пока предыдущее ещё летит, — это не второй параллельный запрос,
а флаг "сохранить ещё раз", который автоматически срабатывает сразу после того, как
текущее сохранение уляжется, уже с актуальным контентом и версией. Сохранения строго
последовательны, поэтому проблема "устаревший ответ перезаписывает свежий" структурно
не может возникнуть.

Статус сохранения — трёхстороннее состояние: "Создание документа…" во время первого
создания, "Сохранено" когда документ синхронизирован с сервером, "Черновик, ещё не
сохранён" при любых незасинканных правках. Любое редактирование сбрасывает статус в
"не сохранено"; успешное сохранение возвращает в "сохранено", пока новых правок нет.
Ошибка сохранения (сеть упала, сервер вернул не-200) показывает inline error banner
прямо под тулбаром — контент в редакторе при этом не трогается и не теряется,
предыдущая ошибка гаснет автоматически при следующем успешном сохранении.

Кнопка "Назад" из редактора возвращает не на лендинг (как было раньше по умолчанию),
а в модалку выбора режима — с уже выбранным типом документа, чтобы не заставлять
пользователя выбирать тип заново.

Повторное открытие ранее сохранённого документа: компонент редактора умеет принимать
существующий documentId вместо создания нового, тянет сохранённый контент и версию с
бэка (GET) и подставляет их в Tiptap вместо пустого черновика. Точки входа в приложении
для этого пока нет — список/история документов вне рамок этой истории (запланирован
отдельной историей #12) — но сам компонент уже умеет это делать и покрыт тестами.

Всё выровнено по мокапам пиксель-в-пиксель (цвета, отступы, иконки-заглушки без
сторонней иконочной библиотеки — по уже устоявшемуся в проекте паттерну).

**Известный незакрытый гэп** (найден тремя независимыми ревью-проходами на последнем
коммите): после повторного открытия документа статус ошибочно показывает "не сохранено",
хотя контент только что подтянут с сервера и не менялся — `hasUnsavedChanges` не
сбрасывается в `false` на этом пути. Плюс если GET при повторном открытии падает —
это уходит только в консоль, без видимого пользователю фидбека, и Save молча
блокируется. Это отдельный bug-task, ещё не заведён.

Selenium-покрытие (визуальная проверка в реальном браузере) для сценариев 3.x-7.x
скипнуто целиком — бэкенд разрабатывается параллельно в другой ветке/сессии и
недоступен здесь для прогона живого приложения. Как только бэкенд станет доступен —
это единственное, что ещё нужно прогнать, чтобы закрыть Frontend-фазу до конца.

## Frontend — расширенный тулбар форматирования (2026-07-16)

Тулбар редактора вырос далеко за пределы исходного набора (bold/italic/списки/заголовки
h1-h2): теперь в нём зачёркивание, цитата (blockquote), горизонтальный разделитель,
инлайн-код, блок кода, undo/redo с реактивным disabled-состоянием и заголовок H3.
Каждая из этих фич цеплялась за одно и то же архитектурное ограничение — схема документа
в Tiptap держит только инлайн-контент (`content: 'inline*'`), поэтому все "блочные" по
природе форматирования (цитата, блок кода, разделитель, H3) пришлось реализовывать не
штатными Node-расширениями Tiptap, а кастомными Mark'ами, которые оборачивают текст
тегом, но не меняют структуру документа. Общая логика "курсор без выделения → применить
форматирование ко всей строке → вернуть курсор на исходную позицию" вынесена в один
переиспользуемый хелпер (`toggleLineMark`), которым пользуются blockquote, code-block и
H3 — к моменту реализации H3 схема уже была настолько обкатана, что фича прошла зелёной
с первой попытки, без единого доп. раунда правок (для сравнения — blockquote и code-block
каждый потребовали по три дополнительных раунда: конфликт со схемой документа, курсор
не возвращался на место после форматирования, и для блока кода — отдельный баг с
"похудением" сохранённого документа при повторном открытии, когда `<pre><code>` тихо
разбирался в обычный инлайн-код).

Два реальных бага найдены и закрыты по ходу: код в тулбаре не открывался в
"неопределённое" состояние при загрузке редактора (кнопки не показывали disabled,
пока Tiptap ещё инициализируется) и стандартный Tiptap-марк для инлайн-кода по
умолчанию вытесняет остальное форматирование (`excludes: '_'`) — если выделить жирный
текст и применить код, жирность тихо пропадает. Второе не входит в acceptance-критерии
сценария и оставлено как отдельная заведённая задача (`tasks/4-bug-code-strips-bold`) —
нужно решение по продукту, ожидаемое это поведение или нет.

Известные, осознанно отложенные и не блокирующие гэпы (задокументированы прямо в
чек-листах сценариев, не потеряны): переключение форматирования "выключить" повторным
кликом при схлопнутом курсоре не покрыто тестами ни для одной из Mark-based фич;
многошаговый undo/redo (несколько правок подряд) проверен только в один шаг; поведение
при курсоре на границе строки (начало/конец) не проверено; совмещение двух Mark'ов на
одном диапазоне текста (например bold+blockquote) не проверено на предмет корректной
сериализации при сохранении/загрузке.
