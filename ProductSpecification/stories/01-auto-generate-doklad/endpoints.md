# Auto-generate: доклад - API Endpoints

| Method | Path | Description |
|--------|------|-------------|
| POST | /api/v1/generations | Submit a new generation request (доклад); enqueues an async job, returns immediately |
| GET | /api/v1/generations/{generation_id} | Get generation status, and document content once `completed` |
| GET | /api/v1/generations | **Planned, not yet implemented** — will list all generations (temporary, unscoped; keyset-paginated). See Scenarios 6.1-6.4, all `[ ]` in progress.md |

## Notes

- `Idempotency-Key` (required request header on POST) — **planned, not yet implemented**.
  The header is not read anywhere in the router/usecase today (Scenario 3.1, `[ ]` in
  progress.md); once built, replaying the same key will return the existing generation
  (`200`, not `201`), never a duplicate. See `01_AutoGenerateDoklad.md` Core Requirements.
- `document_type` is present in the request body now (not hardcoded server-side) so
  stories #2-#4 can reuse this exact endpoint with a different value later; only
  `"доклад"` is accepted today (`422` on anything else).
- No separate "get document" endpoint — the completed document's content is returned
  inline in the `GET /generations/{generation_id}` response, per the MVP-minimal-API rule
  (no split endpoint when one response already carries everything the client needs).
- `GET /generations` is planned as a deliberate, temporary, unauthenticated stand-in for
  history — see `known-debt.md` #4. No `user_id` filter is planned yet because no `User`
  concept exists yet.
- No polling-interval/webhook endpoint — the client polls `GET /generations/{id}` on its
  own schedule (frontend concern, not an API surface).
