## Live-backend integration, 2026-07-16 (register → verify)

Out-of-band task: wire the frontend to the **live** backend rather than the spec. Everything
below was verified by curl against the running stack and by a real headless browser, not read
from `ProductSpecification/api-specs/` — the spec had drifted. Backend runs on **8100**
(`textery/infra/.env`, a DIFFERENT worktree; this worktree's own `infra/.env` says 8001).
`vite.config.ts` proxies `/api` to **8000** by default, which silently 404s — indistinguishable
from a route that does not exist. Run the dev server with
`VITE_API_PROXY_TARGET=http://127.0.0.1:8100`. The default was NOT changed: 8100 belongs to
another worktree's `.env`, and hardcoding it here would violate the no-hardcoded-ports rule.

**Contract, confirmed live (curl, 2026-07-16):**
- `POST /register` → 201 `{user_id, is_verified, email, verification_code, code_expires_at}`;
  409 `{error_code:"EMAIL_ALREADY_REGISTERED", message}`; 422 `{detail:[...]}` when a field is
  missing (Pydantic's envelope, no `error_code`).
- `POST /verify` → 200 `{is_verified:true}`, **idempotent** — replaying the same code returns
  200 again, so a double submit is not an error path. 400 `INVALID_CODE` (not 6 ASCII digits);
  400 `INVALID_OR_EXPIRED_CODE` (wrong / expired / no account / no code — the backend
  deliberately collapses four causes into one code; do not try to tell them apart on the
  client, that is the enumeration oracle 5.2 exists to avoid).
- `POST /login`, `/resend-code`, `/refresh` → **404, not implemented.** Login left untouched
  and NOT mocked into looking functional, per instruction.
- Re-registering an existing account returns 409 whether or not it is verified, and does NOT
  reset `is_verified` — probed explicitly through the whole state machine.

**THIS CORRECTS MY OWN ROUND-7 FINDING (vi), WHICH WAS WRONG.** I filed that `endpoints.md`'s
`{error_code, message}` contradicts a backend that "speaks `{detail}` everywhere", inferring it
from `exception_handlers.py` and the generation routers. Live curl disproves the conclusion:
the **auth** routes really do emit `{error_code, message}`, exactly as `endpoints.md` says. The
`{detail}` shape is real but scoped — Pydantic 422s and the generation/global handlers. The
grep was accurate; the generalization from it was not, and it is the same defect class this
scenario has been cataloguing: a claim about another layer, asserted without a test at that
layer. One curl settled what nine review rounds of reading could not.

**Three shipped frontend bugs, none of which 101 green vitest tests could see** — every one of
them lives exactly where the suite mocks the api module, which is the failure mode the backend
session hit today too (150 green tests missed three bugs that one curl found):
1. `registerApi` sent `{email, password}` with **no `confirm_password`** → 422 → registration
   could never succeed, at all. Fixed: the field is threaded from the form's existing confirm
   input.
2. `applyRegisterError` matched `'EMAIL_ALREADY_EXISTS'`; the backend sends
   `'EMAIL_ALREADY_REGISTERED'`. The duplicate-email message could never render — it fell to
   the generic path. The tests asserted the invented code and passed. Fixed, and the tests now
   assert the real one.
3. `verifyApi` did not exist; `VerifyCodeForm` called only `resendCode` (itself a 404) and
   submitted through `useSubmitPlaceholder`. Route `/verify` rendered `<VerifyCodeForm />` with
   **no props** while the component required `email`, so the chain could not close even with a
   working API. Fixed: new `verifyApi.ts`, real submit, email + code carried in router state.

**The mocked code is the only way in.** `RegisterResult` discarded `verification_code`; no email
is sent. Register now carries `{email, verificationCode}` to `/verify` in **router state, not a
query string** — a query string lands in browser history and server logs, and
`07_Authorization_Notes.md` requires treating the code as a real credential. The verify screen
**shows** the code with a dev-mode notice (`verify-dev-code`) rather than pre-filling it, which
is what that same file asks for (line 20-23: "surface the code directly … so testers aren't
confused"); the input path stays exercised.

**Verified in a real browser against the live stack, not just vitest:** headless Chrome drove
`/register` → filled → submitted → landed on `/verify` → read the dev code off the screen →
typed it → got `verify-success`. Then the row was confirmed in Postgres:
`livec9c3ff3fd7@example.com | t`. Landing on `/verify` is itself the 201 assertion — the
navigate only runs on the success path. Suite after the change: **101/101, tsc clean.**

**QUIRK, cost me a wrong hypothesis — the shared Postgres is wiped under this session.** A
duplicate-email browser run navigated to `/verify` instead of erroring, and the account's
`is_verified` appeared to flip `t → f`. It looked like a backend resetting verification on
re-register. It is not: the row was simply **gone** (`count = 0`) — the parallel backend session
resets the shared volume, exactly the hazard `carryover.md` already records for this host. The
re-register succeeded because it was no longer a duplicate. **So the duplicate-email path is
NOT browser-verified** — curl proves the backend returns 409 with the right code and the unit
tests now assert that code, but the end-to-end render of that message was never observed
against a stable database. Re-run it when the other session is idle.
