  **Two live findings the green usecase/rest tests are structurally blind to — both real, neither
  fixed here (out of this work unit's scope, named rather than left to be discovered later):**
  (1) **`/refresh` returns a byte-identical access token when called twice within the same integer
  second.** JWT `iat`/`exp` are NumericDate (whole seconds), so two issues in one second produce
  identical claims and therefore an identical signature. Measured, not theorized: 12 back-to-back
  refreshes returned **2 distinct tokens** (7x and 5x). Normal stateless-JWT behaviour, no security
  impact — the token is valid and its expiry is correct — but 6.2's title says refresh returns a
  *new* access token, and it does not. Closing it means a `jti` claim. The usecase test cannot see
  this: it asserts against a fake token service. **A frontend test asserting `newToken !== oldToken`
  will flake ~50% of the time.**
  (2) **`code_expires_at` and `access_token_expires_at` serialize in two different formats on the
  same auth surface** — `2026-07-17T07:48:12.459208+00:00` vs `2026-07-17T07:53:14.343092Z`.
  Cause: `RegisterResponseDto.code_expires_at` is typed `str` and built with `.isoformat()`, while
  `LoginResponseDto`'s expiries are typed `datetime` and serialized by pydantic. Both parse in JS
  `new Date()`, so no client breaks today. Also: the DTO expiry carries microseconds the JWT's own
  `exp` claim (whole seconds) does not.

## Security Scenarios (05_Security_Tests.md)

### Scenario 1: Password Hashing
**Pulled forward out of lifecycle order, deliberately** (commit `7c28c08`). This phase normally
runs after Frontend, but plaintext credentials were already being persisted to real rows and that
is the one defect in this codebase that cannot be fixed retroactively. It also had to land before
5.x: writing login against plaintext and swapping the scheme later means redoing login and a
migration twice. Trimmed cycle (`decisions/trimmed-tdd-cycle-sprint-1-decision.md`), except the
tests, which were not optional here — the existing suite encodes plaintext persistence and goes
red on the change either way.
- [S] red-acceptance — no acceptance test; the DB-row inspection the Gherkin describes is a
  db-adapter concern, covered by `adapters/security` tests + a live check (`$2b$12$...` confirmed
  in Postgres)
- [x] design — `PasswordHasher` port (usecase) + `BcryptPasswordHasher` in a **new
  `backend/adapters/security` module**. `bcrypt` is third-party, so it cannot live in the domain
  next to `Password` (domain has no dependencies); the `SystemClock` precedent of an inline
  usecase implementation does not apply — that one is stdlib-only. Port is a **required**
  constructor arg with **no null-object fallback**: a default returning the plaintext would
  persist real credentials in the clear with every test still green — the same shape as the null
  repository that silently discarded accounts in 1.5.
- [x] red-usecase — `assert_password_hashed_from_normalized_form`. Scenario 2.7's assertion was
  **carried over, not deleted**: it asserted the persisted value *was* the NFC form, which hashing
  makes false by design; the claim it actually protects (the hash is computed from the NFC form,
  never the raw string) is now what it pins. `FakePasswordHasher` used in usecase tests — real
  bcrypt is ~100ms/call by design and would dominate the suite.
- [x] green-usecase — `RegisterUser` hashes `password_value_object.value` (NFC-normalized), so an
  NFD and an NFC submission of the same password hash identically — which is exactly what 2.7
  exists to guarantee.
- [x] adapters-discovery — new module needs: `tests/conftest.py` `sys.path` wiring (the repo's
  convention), `container.py` injection, `bcrypt==5.0.0` in `requirements.txt`, and **`main.py`'s
  hand-built `sys.path`** (verified by importing the app, not assumed). Dockerfile needs no change
  — it already `COPY`s `backend/adapters` wholesale.
- [x] green-acceptance — 7 adapter tests on real bcrypt; live Postgres confirms `$2b$12$...`.
  **Non-obvious find:** bcrypt truncates at 72 bytes, but `Password` allows 128 code points and
  **scenario 2.4d pins a 252-UTF-8-byte password as valid with a 201** — truncating would make two
  different long passwords interchangeable at login, rejecting would 500 a valid request. Input is
  SHA-256 pre-hashed + base64-encoded (the passlib `bcrypt_sha256` construction) so every input is
  44 bytes; base64 keeps a NUL byte out of the digest, which bcrypt treats as a terminator. A test
  pins the hole directly: two passwords sharing their first 100 chars are not interchangeable.
