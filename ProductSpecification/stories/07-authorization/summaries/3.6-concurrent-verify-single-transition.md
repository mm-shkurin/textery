# Scenario 3.6 ‚Äî Concurrent verify requests produce exactly one transition

## green-adapter db (2026-07-20)

**Quirk:** `SqlAlchemyAccountRepository.transition_to_verified` uses a Core bulk `update()`, which bypasses the ORM identity map.
**Where:** `backend/adapters/db/src/access/auth/account_storage.py`.
**Implication:** In production `VerifyAccount.execute` calls `find_by_email` (an ORM load) in the same session first, so after the bulk UPDATE the loaded `AccountModel.is_verified` is stale ‚Äî any later same-transaction read of it must re-read or use `synchronize_session`, not trust the in-memory value.

## design (2026-07-20)

**Decision:** A zero-row result from `transition_to_verified`/`transition_to_consumed` returns `False` (idempotent success), never raises ‚Äî the opposite of `generation_storage.py:91`'s zero-row‚Üíraise.
**Why:** The losing concurrent racer must resolve to success, or the usecase's broad `except` turns it into `VerificationFailedException` and breaks the already-committed both-200 `test_verify_concurrent_acceptance.py` pin.
**Where applied:** the two conditional-UPDATE storage methods and `VerifyAccount._apply_verification`.
