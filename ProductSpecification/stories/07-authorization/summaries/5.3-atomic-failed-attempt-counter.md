# Scenario 5.3: Failed-attempt counter increments atomically across concurrent failures

## design (2026-07-21)

**Surprise:** the ADR first named `f6a7b8c9d0e1` (accounts_email_unique) as the alembic head based on file mtime, but that revision already has a child (`a7b8c9d0e1f2`); the true head is `e1f2a3b4c5d6` (generations_updated_at).
**Why:** newest-by-mtime (`ls -t`) is not revision order — migrations get touched out of chain order; only `alembic heads` reports the real terminal revision.
**Impact:** basing a new migration's `down_revision` on a non-head revision silently creates a SECOND head and breaks `alembic upgrade head`; always resolve the head with `alembic heads`, and give the new column a `server_default` so `op.add_column NOT NULL` backfills existing rows.

## red-usecase (2026-07-21)

**Surprise:** giving `LoginUser` an optional `unit_of_work=None → NullUnitOfWork()` means that if `create_login_user` wiring forgets to pass a real `SqlAlchemyUnitOfWork`, the green `commit()` becomes a silent no-op — the increment SQL is issued on the session but never committed and is discarded on `session.close()`.
**Why:** `NullUnitOfWork.commit()` is a no-op by design, and the wiring's `finally: session.close()` rolls back the uncommitted UPDATE.
**Impact:** the failed-attempt counter would stay 0 in production with zero visible symptom (acceptance is `[S]`); the wiring pin must assert BOTH that the UoW is real (not None) AND shares the account repo's session — session identity alone won't catch an unwired UoW.
