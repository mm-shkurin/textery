# Scenario 3.4 — Re-submitting an already-consumed code is idempotent

## red-usecase (2026-07-20)

**Surprise:** The "idempotent" replay was never an internal no-op — a second `/verify` with the same code re-ran the whole tail: `consume()` overwrote `consumed_at` with a fresh timestamp and a second `commit()` fired.
**Why:** `find_active_by_account_id` returns the latest code regardless of consumption, so `matches()` still held and nothing short-circuited before `_apply_verification`.
**Impact:** The invariant is invisible at HTTP (both answers are 200), so it can only be pinned negatively at the usecase layer (save/commit counts frozen, `consumed_at` preserved) — a counting-only guard goes green on the bug.
