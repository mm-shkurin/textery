## red-selenium (2026-07-16)

**Mistake:** Concluded `POST /api/v1/auth/login` did not exist because curl returned 404, and deferred the step on that basis.
**Why wrong:** The probe hit a container built before the route landed — "not in the image" is not "not in the code", and the 404 was an honest answer about a stale artifact.
**Correct location/approach:** On a 404/500 from an auth route, `docker compose up -d --build backend` first, then conclude; `GET /login` → 405 and `openapi.json` both confirm a route exists.

## red-frontend-api (2026-07-16)

**Quirk:** Every component test mocks the api module wholesale, so the whole suite stayed green against an invented wire contract — three shipped bugs (missing `confirm_password` → 422, an `EMAIL_ALREADY_EXISTS` code nothing emits, camelCase token fields the wire never sends) were invisible to 101 passing tests.
**Where:** `frontend/src/features/auth/components/__tests__/*` mock `../../api/*`; only `api/__tests__/loginApi.test.ts` uses real fetch via `vi.stubGlobal`.
**Implication:** A green unit suite is not evidence that an api client matches the wire; only a real-fetch test or a browser run against the live stack is.

## discussion (2026-07-16)

**Decision:** Auth tokens live in `sessionStorage`, not localStorage or memory+refresh.
**Why:** The backend returns tokens in the response body rather than an httpOnly cookie, so the client must hold them and no XSS-proof option exists — sessionStorage kills a stolen token with the tab, survives F5, and needs no refresh logic to get wrong.
**Where applied:** `frontend/src/features/auth/utils/authSession.ts`, consumed by `LoginForm`.

## discussion (2026-07-16)

**Decision:** The landing stays public; everything behind it (type → mode → workspace) requires a session.
**Why:** The landing is the shopfront and gating it would gate marketing; the user's rule was that an anonymous visitor sees the landing and an authenticated one can generate.
**Where applied:** `DocumentGenerationFlow` in `frontend/src/App.tsx` — the CTA is the only door into the workspace, which has no URL of its own.

## green-selenium (2026-07-16)

**Quirk:** Selenium assertions on in-flight UI state stopped being observable once the real APIs replaced `useSubmitPlaceholder`'s 500ms delay — the backend answers in milliseconds, so the disabled/loading window closes before the poll sees it.
**Where:** `acceptance/tests/frontend/auth/test_login_submit_disabled_while_in_flight_acceptance.py`, `..._loading_indicator_...`, `test_verify_confirm_disabled_while_in_flight_...` — all three fail; baselined by stashing, they fail without any of today's changes.
**Implication:** Any scenario asserting a state DURING a request needs a controllable delay (a stubbed slow response); assertions on settled states are unaffected.

## red-frontend (2026-07-16)

**Quirk:** jsdom's `window.location` survives between tests in a file, and `App` renders its own `BrowserRouter`, so one test navigating away leaves every later render starting at that URL.
**Where:** `frontend/src/__tests__/App.test.tsx` — `window.history.pushState({}, '', '/')` in `beforeEach`.
**Implication:** Any test that triggers navigation must reset the URL, not just the session, or it silently corrupts its neighbours.
