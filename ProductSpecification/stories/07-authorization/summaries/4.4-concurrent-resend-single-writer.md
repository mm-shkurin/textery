# Scenario 4.4: Concurrent resend requests do not both succeed within the cooldown

## green-adapter db (2026-07-21)

**Surprise:** a two-racing-session db test with a NO-OP lock still passed `exactly-one` — asyncio's cooperative scheduler serialized the two coroutines on the single event loop (racer 2's read landed after racer 1's commit), so the lost-update never occurred.
**Why:** `asyncio.gather` interleaves only at `await` points; without a suspension between the read and the commit, one coroutine runs its whole critical section before the other starts.
**Impact:** the DSL needs an `asyncio.sleep(_RACE_WINDOW_SECONDS)` between the read and the commit to force the interleave a real `SELECT … FOR UPDATE` closes — otherwise the falsification (no-op lock → double-issue) is masked and the test cannot prove the lock has teeth.

## red-usecase (2026-07-21)

**Surprise:** the exactly-one-under-concurrency guarantee is split across a db-adapter race test (hand-orchestrated lock→read→insert) and a usecase test whose Fake lock is a single-threaded no-op — neither drives real `ResendCode.execute` through real repos under concurrency.
**Why:** serialization is a property of the FOR UPDATE lock and the insert sharing ONE session/transaction (`create_resend_code`), which only the composition root wires — Fakes can't model it and the db DSL bypasses `execute`.
**Impact:** the load-bearing guard is a wiring test asserting the account repo, code repo, and UoW share one `AsyncSession`; a future refactor giving any collaborator its own session silently un-serializes resend with every other test still green.
